What React Actually Is

You know how in vanilla JS you write this when data changes:

// Vanilla JS — you manage the DOM manually
document.getElementById('count').textContent = newCount;
document.getElementById('btn').classList.toggle('active', newCount > 0);

React's job is to do that for you — you just describe what the UI should look like given the current data, and React figures out what to update in the DOM.

Component
A function that returns UI. Like a reusable HTML block with logic built in.
Props
Data passed into a component from outside. Like HTML attributes, but for your components.
State
Data that lives inside a component. When it changes, React re-renders the UI automatically.
JSX
HTML-like syntax inside JavaScript. Compiled to regular JS by the build tool.

JSX & Components Day 1

The foundation of everything in React

What Is JSX? Concept

JSX looks like HTML but it's not. It's JavaScript that describes what the DOM should look like. There are a few key differences from HTML:

HTML

<div class="card">
  <img src="photo.jpg">
  <label for="name">Name</label>
</div>

JSX

<div className="card">
  <img src="photo.jpg" />
  <label htmlFor="name">Name</label>
</div>
  • class → className (class is a reserved JS keyword)
  • for → htmlFor
  • Self-closing tags MUST close: <img /> not <img>
  • You can only return one root element per component
  • Use {} to embed any JavaScript expression

Your First Component Concept

A component is just a function that returns JSX. Name it with a capital letter (this is mandatory — lowercase = HTML tag, uppercase = your component).

UserCard.jsx
// A component is a function that returns JSX
function UserCard() {
  const name = "Sarah Ahmed";
  const isOnline = true;

  return (
    <div className="card">
      <h2>{ name }</h2>
      <p>{ isOnline ? "🟢 Online" : "⚫ Offline" }</p>
      <p>Joined: { new Date().getFullYear() }</p>
    </div>
  );
}

// Always export your component
export default UserCard;
💡 Why Return Parentheses?
The parentheses around JSX are required when it spans multiple lines. Without them, JavaScript's automatic semicolon insertion will silently break your code.

Using a Component

App.jsx
import UserCard from './UserCard';

function App() {
  return (
    <div>
      <UserCard />   {/* Use it like an HTML tag */}
      <UserCard />   {/* Reuse it! */}
    </div>
  );
}

export default App;

Props Day 1

How you pass data into components

What Are Props? Concept

Props are how you pass data into a component from outside. Think of them exactly like HTML attributes — except you define them yourself.

💡 Why Props?
Without props, every component is hardcoded and useless for reuse. Props are what make components generic and reusable — the difference between a component that shows "Sarah" forever and one that shows whatever name you give it. 
UserCard.jsx
// ✅ Accept props as the first argument (destructure for clarity)
function UserCard({ name, role, avatar, isOnline }) {
  return (
    <div className="card">
      <img src={avatar} alt={name} />
      <h2>{name}</h2>
      <p>{role}</p>
      <span>{isOnline ? "Online" : "Offline"}</span>
    </div>
  );
}

export default UserCard;
App.jsx
import UserCard from './components/UserCard';

function App() {
  return (
    <div>
      {/* Pass data just like HTML attributes */}
      <UserCard
        name="Sarah Ahmed"
        role="Frontend Developer"
        avatar="/avatars/sarah.jpg"
        isOnline={true}
      />

      <UserCard
        name="Ali Khan"
        role="UI Designer"
        avatar="/avatars/ali.jpg"
        isOnline={false}
      />
    </div>
  );
}

Default Props Concept

// Set defaults directly in the destructure
function Button({ label = "Click me", variant = "primary", onClick }) {
  return (
    <button
      className={`btn btn-${variant}`}
      onClick={onClick}
    >
      {label}
    </button>

The children Prop Concept

React has a special prop called children — it's whatever you put between your component's opening and closing tags.

// This is a layout wrapper component
function Card({ children, title }) {
  return (
    <div className="card">
      <h3>{title}</h3>
      <div className="card-body">
        {children}  {/* Renders whatever is inside <Card>...</Card> */}
      </div>
    </div>
  );
}

// Usage:
<Card title="User Info">
  <p>Name: Sarah</p>
  <p>Role: Developer</p>
</Card>

State with useState Day 1

Data that lives inside a component and drives re-renders

The Problem State Solves Concept

Props are read-only — you can't change them from inside a component. When you need data that changes over time (a counter, a toggle, user input), you need state.

💡 The Key Rule
When state changes, React automatically re-renders the component with the new value. You never touch the DOM yourself. This is the entire value proposition of React. 
Counter.jsx
import { useState } from 'react';  // Always import from react

function Counter() {
  // useState(initialValue) returns [currentValue, setterFunction]
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>-</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}

State with Objects Concept

When your state is an object, never mutate it directly. Always create a new object using the spread operator ...:

const [user, setUser] = useState({
  name: "",
  email: "",
  age: 0
});

// ❌ WRONG — mutating state directly (won't trigger re-render)
user.name = "Sarah";
setUser(user);

// ✅ CORRECT — create a new object, spread old values, override what changed
setUser({ ...user, name: "Sarah" });

// ✅ CORRECT — using functional update (when new state depends on old state)
setUser(prev => ({ ...prev, name: "Sarah" }));

State with Arrays Concept

const [items, setItems] = useState([]);

// Add an item
setItems(prev => [...prev, newItem]);

// Remove an item
setItems(prev => prev.filter(item => item.id !== targetId));

// Update one item
setItems(prev => prev.map(item =>
  item.id === targetId ? { ...item, done: true } : item
));

Where To Put State? Concept

📍 The Rule of Thumb
Put state in the lowest common ancestor of all components that need it. If only one component needs the state, keep it there. If a sibling needs it too, lift it up to the parent.

Events & Forms Day 1

Handling user interaction in React

Event Handling Concept

Events in React are camelCase and take a function, not a string:

HTML

<button
  onclick="handleClick()"
>Click</button>

JSX

<button
  onClick={handleClick}
>Click</button>
function MyButton() {
  const handleClick = (e) => {
    e.preventDefault();  // Works the same as vanilla JS
    console.log("clicked!", e.target);
  };

  return <button onClick={handleClick}>Click Me</button>;
}

Controlled Forms Concept

In React, form inputs are controlled — you store the input's value in state, and feed it back into the input. React owns the value, not the DOM.

LoginForm.jsx
import { useState } from 'react';

function LoginForm() {
  const [formData, setFormData] = useState({
    email: "",
    password: ""
  });

  const handleChange = (e) => {
    // e.target.name matches the input's name attribute
    setFormData(prev => ({
      ...prev,
      [e.target.name]: e.target.value
    }));
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log("Submitted:", formData);
    // Make API call here
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        name="email"
        value={formData.email}
        onChange={handleChange}
        placeholder="Email"
      />
      <input
        type="password"
        name="password"
        value={formData.password}
        onChange={handleChange}
        placeholder="Password"
      />
      <button type="submit">Login</button>
    </form>
  );
}

Lists & Keys Day 1

Rendering dynamic collections of data

Rendering Lists Concept

Use .map() to convert an array of data into an array of JSX elements:

const users = [
  { id: 1, name: "Sarah", role: "Developer" },
  { id: 2, name: "Ali", role: "Designer" },
  { id: 3, name: "Zara", role: "Manager" },
];

function UserList() {
  return (
    <ul>
      {users.map(user => (
        {/* key must be unique and stable — use an ID, not the index */}
        <li key={user.id}>
          <strong>{user.name}</strong> — {user.role}
        </li>
      ))}
    </ul>
  );
}
⚠ Common Mistake: Using Index as Key

key={index} causes subtle bugs when list items are deleted, reordered, or filtered. Always use a unique ID from your data. If your data has no ID, use a library like nanoid to generate one when creating items.

Conditional Rendering Concept

function UserList({ users, isLoading }) {
  // Early return for loading state
  if (isLoading) return <p>Loading...</p>;

  // Handle empty state
  if (users.length === 0) return <p>No users found.</p>;

  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>
          {user.name}
          {/* &&  renders right side only if left side is truthy */}
          {user.isAdmin && <span>👑 Admin</span>}
        </li>
      ))}
    </ul>
  );
}

Figma → React Day 2

A repeatable process for converting any design to components

The 5-Step Process Concept

  1. 1
    Inspect & Extract Values

    Open Figma. Click on elements. Copy exact values: font-size, font-weight, color hex codes, border-radius, spacing. Use Dev Mode if available. Define these as CSS variables in a variables.css or in your CSS file upfront.

  2. 2
    Identify Repeating Patterns

    Scan the whole design. Circle anything that appears more than once — those are your components. A button that appears 8 times should be one Button component, not 8 separate buttons.

  3. 3
    Break Into Component Hierarchy

    Draw boxes around groups of UI. Start at the page level → sections → cards → small atoms. Give each box a component name. Bigger = parent, smaller = children.

  4. 4
    Build Bottom-Up

    Start with the smallest, most reusable pieces (Button, Badge, Avatar) and work outward to larger containers. Never start with the whole page.

  5. 5
    Wire in Real Data

    Replace hardcoded text with props. Any text you see in the design that would change between instances becomes a prop. Then pass actual data from a data file or API.

Breaking UI Into Components Day 2

The skill that separates juniors from mid-level developers

The Atomic Design Mental Model Concept

Think of your UI in 3 levels. This isn't a rigid rule — it's a thinking tool:

Atoms (tiny, reusable)
Button, Input, Badge, Avatar, Spinner, Icon. Can't be broken down further. Used everywhere.
Molecules (composed)
SearchBar (Input + Button), UserCard (Avatar + name + role), FormField (Label + Input + Error).
Organisms (sections)
Header, Sidebar, ProductGrid, UserList, HeroSection. Full self-contained sections of a page.
Pages
HomePage, ProfilePage, DashboardPage. Compose organisms. Handle routing and top-level data fetching.

Worked Example: E-commerce Header Concept

Given this Figma header: Logo | Search Bar | Cart icon with count | User avatar | Nav links

// How to break it down:

<Logo />                      // just the brand mark

<IconButton icon badge />      // icon with optional count

// Molecules (combine atoms)
<SearchBar onSearch />         // input + search button + results
<NavLinks links />             // list of navigation links
<CartButton count />           // IconButton + cart count
<UserMenu user />              // Avatar + dropdown menu

// Organism: the full header
function Header({ user, cartCount }) {
  return (
    <header className="header">
      <Logo />

The Rule for Splitting Concept

Split a component when any of these are true:

  • It can be reused in 2+ places
  • It has its own local state that doesn't affect others
  • The file is getting long (over ~80 lines of JSX is a code smell)
  • It represents a visually distinct "thing" in the design

CSS Modules — The Professional Way to Style Concept

Avoid global CSS. Use CSS Modules — every component gets its own scoped styles, so class names never clash.

Button.module.css
.btn {
  padding: 10px 20px;
  border-radius: 8px;
  border: none;
  cursor: pointer;
  font-size: 14px;
  font-weight: 500;
}

.primary {
  background: var(--color-primary);
  color: white;
}

.secondary {
  background: transparent;
  border: 1px solid var(--color-primary);
  color: var(--color-primary);
}
Button.jsx
import styles from './Button.module.css';

function Button({
  label,
  variant = "primary",
  onClick,
  disabled
}) {
  return (
    <button
      className={
        `${styles.btn} ${styles[variant]}`
      }
      onClick={onClick}
      disabled={disabled}
    >
      {label}
    </button>
  );
}

export default Button;
💡 Why CSS Modules?
When two developers on a team both write a class called .card, they overwrite each other's styles. CSS Modules automatically generate unique class names like Button_btn__xK3p, so this never happens.

useEffect — The Basics Day 2

Running code in response to renders and data changes

What Is useEffect? Concept

useEffect runs code after a component renders. Use it for side effects — things that happen outside of rendering: fetching data, setting up event listeners, updating the document title.

import { useState, useEffect } from 'react';

function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // This runs AFTER the component renders
    const fetchUser = async () => {
      const res = await fetch(`/api/users/${userId}`);
      const data = await res.json();
      setUser(data);
      setLoading(false);
    };

    fetchUser();
  }, [userId]); // ← Dependency array: re-runs when userId changes

  if (loading) return <p>Loading...</p>;

  return <div>{user.name}</div>;
}

The Dependency Array Concept

No array: useEffect(() => {})
Runs after EVERY render. Rarely what you want.
Empty array: useEffect(() => {}, [])
Runs ONCE on mount. Perfect for initial data fetch.
With deps: useEffect(() => {}, [id])
Runs on mount AND whenever id changes.
Return a cleanup: return () => {}
Runs before the component unmounts or before the effect re-runs. Use for timers, subscriptions.
⚠ Beginner Trap: Infinite Loop

If you update state inside useEffect without a dependency array, it will loop forever — state update → re-render → effect runs → state update → ...

// ❌ INFINITE LOOP — no dependency array
useEffect(() => {
  setCount(count + 1);
});

// ✅ Runs only once
useEffect(() => {
  setCount(1);
}, []);

import { useState } from 'react';
import Header from '../components/layout/Header';
import FilterBar from '../components/shared/FilterBar';
import ProductCard from '../components/shared/ProductCard';
import { products } from '../data/products';

function ShopPage() {
  const [cart, setCart] = useState([]);
  const [category, setCategory] = useState("all");
  const [sortBy, setSortBy] = useState("default");

  const addToCart = (product) => {
    setCart(prev => {
      const exists = prev.find(item => item.id === product.id);
      if (exists) {
        // Increase quantity if already in cart
        return prev.map(item =>
          item.id === product.id
            ? { ...item, qty: item.qty + 1 }
            : item
        );
      }
      return [...prev, { ...product, qty: 1 }];
    });
  };

  const cartCount = cart.reduce((sum, item) => sum + item.qty, 0);

  const filtered = products
    .filter(p => category === "all" || p.category === category)
    .sort((a, b) => {
      if (sortBy === "price-asc") return a.price - b.price;
      if (sortBy === "price-desc") return b.price - a.price;
      return 0;
    });

  return (
    <div>
      <Header cartCount={cartCount} />
      <main className="container">
        <FilterBar
          category={category}
          onCategoryChange={setCategory}
          sortBy={sortBy}
          onSortChange={setSortBy}
        />
        <div className="product-grid">
          {filtered.map(product => (
            <ProductCard
              key={product.id}
              {...product}
              onAddToCart={() => addToCart(product)}
            />
          ))}
        </div>
      </main>
    </div>
  );
}

export default ShopPage;

Common Beginner Mistakes

Avoid these and you'll look like you've been doing React for months.

1. Mutating State Directly Mistake

// ❌ WRONG — mutating the state array directly
const [items, setItems] = useState(['a', 'b']);
items.push('c');         // React won't notice this change!
setItems(items);        // Same array reference → no re-render

// ✅ CORRECT — always create new array
setItems([...items, 'c']);

2. Calling setState in Render Mistake

// ❌ WRONG — this causes an infinite render loop
function MyComponent() {
  const [count, setCount] = useState(0);
  setCount(1);  // ← called during render!
  return <div>{count}</div>;
}

// ✅ State updates go in event handlers or useEffect

3. Not Using a Key on Lists Mistake

// ❌ WRONG — no key, or using index as key
items.map((item, index) => <Card key={index} />)

// ✅ CORRECT — unique stable ID
items.map(item => <Card key={item.id} />)

4. Forgetting to Export Mistake

// ❌ WRONG — component defined but never exported
function MyCard() { ... }
// No export — importing this gives "undefined"!

// ✅ CORRECT
export default MyCard;   // named default export
export { MyCard };       // OR named export

5. Putting Fetch Logic Inside JSX Mistake

// ❌ WRONG — async in render, will crash
return (
  <div>{ await fetchUser() }</div>
);

// ✅ CORRECT — fetch in useEffect, render the state
useEffect(() => {
  fetchUser().then(data => setUser(data));
}, []);
return <div>{user?.name}</div>;

6. Too Many Props ("Prop Drilling") Mistake

⚠ Watch For This
If you find yourself passing the same prop through 3+ levels of components (A → B → C → D just to use it in D), that's prop drilling. The solution at your level is to restructure your components so D is closer to where the data lives. Full fix later: React Context.

7. className vs class Mistake

// ❌ HTML attribute in JSX — this silently fails
<div class="card">

// ✅ JSX attribute
<div className="card">
